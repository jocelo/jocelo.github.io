Definition

The queue is a data structure, even more specific, is an abstract data type that implements the FIFO principle.

FIFO stands for First Input First Output; in other words, the first element that is inserted is the first element that gets out.

A quick analogy that helps with understanding this concept is: Imagine the line of people outside the movie theater... The first person that got into the line is the first person that gets inside the theater. Given that everybody is respectful.

Basic operations that you can perform on a queue are pretty similar as those you can do on a stack: insert of a new element/data and the retrieve of an element/data. These two operations are usually called push and shift respectively.

We said usually, because they might be called different in another programming language, but the idea is the same.

To implement this data type, as usual, it depends on the programming language of our choice, and it must likely that you will have these operations already part of the methods available to from the base code, so you just have to learn how to use them. These methods are part of the array object (or a list on python).

The operations you perform on a queue are in the different ends of the queue: retrieving is performed at the beginning of the queue and insertion is made at the bottom of the queue.

Complexity analysis (big O)

The queue is an abstract data type with tons of uses in coding. On it's basic implementation, the complexity for read and write is of linear time O(n); where n is the size of the queue. We get this time complexity because whenever we insert or retrieve an element from the queue, we neeed to shift all the remainder elements from the list. This is, we need to shift all elements to the right when inserting a new element, and shift all elements to the left, when retrieving an element.

Note: The term read used above, is used in context of the shift operation. This usage of the term read must not be confused with other read operations executed in the queue ( peek is the most common term for actually reading the top of the queue... just read, not extracting a value ). Speaking of which, the peek operation has a constant time complexity O(1) because we are only reading the element, we do not have to shift the position of any elements. There's another way to work with queues that help us reduce the time complexity, this discussion was left out of this article to reduce the complexity, however there are some programing languages that have some modules/objects that are part of the code base and let us work with queues in a more efficient way. One example is python (python2.7 and above). Python has an object called deque (double-ended queue) that is part of the collections module; this one offers a constant time complexity O(1) in insert and retrieve operations. You can read more about this object in the official documentation.
